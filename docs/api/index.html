API RPN — Aperçu

L'API JSON de RPN permet de piloter le site à distance : créer des articles et des questionnaires (quiz), sans passer par l'interface web. Elle sert par exemple à demander à un assistant de rédiger un cours et de le publier directement, avec son quiz associé.

Base URL (production) : https://bokonzi.com/rpn
Authentification : clé API dans l'en-tête X-API-Key

En 30 secondes

# 1. Tester que l'API répond (public)
curl https://bokonzi.com/rpn/api/ping
# → {"ok":true,"service":"RPN API","date":"…"}

# 2. Créer un article (clé requise)
curl -X POST https://bokonzi.com/rpn/api/article \
  -H "X-API-Key: VOTRE_CLE" -H "Content-Type: application/json" \
  -d '{"title":"Mon article","content":"<p>Bonjour</p>"}'

# 3. Créer un quiz lié à l'article 15
curl -X POST https://bokonzi.com/rpn/api/quiz \
  -H "X-API-Key: VOTRE_CLE" -H "Content-Type: application/json" \
  -d '{"title":"Mon quiz","article_id":15,"questions":[
        {"body":"2+2 ?","type":"single","options":[
          {"label":"4","correct":true},{"label":"5","correct":false}]}]}'
La clé API — référence : réglage api_key (régénérable depuis admin/settings) ; à défaut, la constante API_KEY de config/config.php. Régénérer la clé révoque immédiatement l'ancienne.

1. Comment l'API fonctionne

Chaque appel passe par le routeur (index.php + config/routes.php) puis le contrôleur ApiController, qui hérite du socle commun ApiKernel. Le cycle d'une écriture est toujours le même :

Requête HTTP
   │
   ▼
1. readInput()   → lit le corps (JSON ou formulaire)
2. requireKey()  → vérifie la clé API   ──►  401 si absente / invalide  (STOP)
3. validation    → champs requis ?       ──►  422 si non  (STOP, rien créé)
4. nettoyage     → strip_tags / Html::clean (anti-XSS)
5. écriture      → modèles en requêtes préparées (PDO)
   │
   ▼
Réponse JSON  { ok:true, … }  (201 Created)

Authentification

La clé est transmise (par priorité) : en-tête X-API-Key (recommandé), champ key du corps JSON, $_POST, ou ?key=. Comparaison par hash_equals (temps constant).

Format des réponses

CasForme
Succès{ "ok": true, … } + code 200 (ping) ou 201 (création)
Erreur{ "ok": false, "error": "message" } + code 401 / 422 / 404 / 500

2. Routes (« liens ») & lien article ↔ quiz

MéthodeLien (URL)ActionAuth
GET/rpn/api/pingApiController::ping— public
POST/rpn/api/articleApiController::createArticleclé API
POST/rpn/api/quizApiController::createQuizclé API
Méthode stricte : api/article et api/quiz n'existent qu'en POST. Un GET dessus → page introuvable (voulu). Et les URLs sont sans .php : /rpn/api/article, pas …article.php.

Le lien article ↔ quiz

Un quiz n'appartient pas à un article : le lien passe par une table de jointure, ce qui permet d'associer un quiz à plusieurs articles (ou aucun).

article_quizzes ( article_id , quiz_id , position )
Méthode (modèle Article)Rôle
quizIds($articleId)ids des quiz déjà liés
setQuizzes($articleId, $ids)(re)définit la liste des quiz liés
quizzesFor($articleId)quiz publiés d'un article (affichage)

Quand POST /rpn/api/quiz reçoit un article_id valide, il crée le quiz puis l'ajoute aux liens existants via setQuizzes (les liens en place sont conservés). La page de l'article affiche alors automatiquement le bloc « Réponds au questionnaire ci-dessous ».

Associer un quiz à n'importe quel article : il suffit de mettre son id dans la requête — { "title":"…", "article_id": 11, "questions":[…] }.

3. Pourquoi l'API peut bloquer

CodeerrorCauseSolution
401Clé API manquante ou invalide.Pas de clé, mauvaise clé, ou clé révoquéeEnvoyer la bonne clé dans X-API-Key. Si régénérée, l'ancienne ne marche plus.
422Champs requis : title et content.Article sans titre/contenuFournir title et content non vides.
422Champs requis : title et questions[].Quiz sans titre ou sans questionsFournir title et ≥ 1 question.
422Aucune question valide …Toutes les questions invalidesChaque question : énoncé + ≥ 2 options + ≥ 1 correct.
404(page introuvable)Mauvaise URL, ou GET sur route POSTURL exacte sans .php, en POST.
500Erreur serveur …Base/​table/​droits/​déploiement partielVoir ci-dessous.
Astuce 401 vs clé OK : envoie un POST avec la clé mais sans les champs. Si tu obtiens 422 (pas 401), c'est que la clé est bonne.

401 — le blocage le plus fréquent

Vérifie dans l'ordre : (1) en-tête bien envoyé ; (2) bonne clé (réglage api_key sinon API_KEY) ; (3) pas régénérée entre-temps (révoque l'ancienne) ; (4) clé copiée en entier (une clé tronquée → toujours 401).

500 — erreur serveur

  • Base de données indisponible ou identifiants config.php erronés.
  • Table manquante (article_quizzes, quizzes…) → importer database/database.sql.
  • ApiKernel introuvable après une mise en ligne partielle : ApiController hérite d'ApiKernel → déployer les deux fichiers ensemble.
  • Droits d'écriture insuffisants sur uploads/ ou storage/.
Ce qui n'est PAS un blocage : la page d'un quiz qui demande de se connecter (réponses par membre) ; un quiz en brouillon (active=0) non affiché sous l'article (quizzesFor() ne montre que les quiz publiés).

4. Sécurité

On ne peut RIEN modifier sans la clé

  • requireKey() est appelé en premier dans chaque écriture → 401, aucune ligne créée.
  • Comparaison par hash_equals() (temps constant, anti-timing-attack).
  • Clé régénérable depuis admin/settings ; régénérer révoque l'ancienne.
  • ping est le seul point public et ne modifie rien.
  • Les écritures par l'interface web sont protégées par la session (canal indépendant).

Pas d'injection SQL

  • 100 % requêtes préparées (PDO + paramètres liés) dans les modèles.
  • Seule interpolation : LIMIT $limit dans Article::announcements() — mais $limit est un int typé et borné (max(1, min(20, …))). Non injectable.
  • ApiKernel ne construit aucune requête SQL : il délègue aux modèles.

Pas d'injection HTML / XSS

  • Contenu HTML des articles filtré par Html::clean (liste blanche de balises sûres ; script/style/iframe supprimés ; attributs retirés sauf href sûr).
  • Texte des quiz (titre, énoncés, options, description) stocké sans balise (strip_tags) et échappé (htmlspecialchars) à l'affichage.

Checklist de revue

☑ Écriture impossible sans clé valide (401) — testé  ·  ☑ Comparaison à temps constant (hash_equals)  ·  ☑ Requêtes préparées partout  ·  ☑ HTML filtré / texte quiz échappé  ·  ☑ Validation avant écriture (422 sans effet de bord)

5. Architecture MVC — socle ApiKernel

Avant, les actions createArticle et createQuiz recopiaient la même plomberie (parsing, clé, réponse JSON, URL, téléchargement). Désormais tout est regroupé une seule fois dans app/core/ApiKernel.php, dont héritent les contrôleurs d'API.

ApiKernel (app/core/)          ← logique transversale, une seule fois
   ▲
   │ extends
ApiController (app/controllers/)
   ├── ping()
   ├── createArticle()          ← uniquement la logique métier
   └── createQuiz()
Méthode d'ApiKernelRôle
readInput()lit une fois le corps JSON (idempotent)
field($k,$def)champ : corps JSON → $_POST → défaut
text($k,$max)texte propre : strip_tags + trim + troncature
json() / fail()réponse JSON / erreur normalisée
requireKey()sécurité : exige une clé valide (401 sinon)
absoluteUrl()URL absolue vers une route interne
downloadImage()télécharge + enregistre une image distante

Ajouter un endpoint devient trivial

class ApiController extends ApiKernel
{
    public function createMachin(): void
    {
        $this->readInput();
        $this->requireKey();                 // sécurité héritée
        $nom = $this->text('nom', 100);      // parsing hérité
        if ($nom === '') { $this->fail('nom requis', 422); }
        // $id = Machin::create([...]);      // modèle (requête préparée)
        $this->json(['ok' => true, 'id' => $id,
            'url' => $this->absoluteUrl('machin', ['id' => $id])], 201);
    }
}

Puis une ligne dans config/routes.php : $router->post('api/machin', 'ApiController', 'createMachin');

Fichiers concernés

6. Tests

Tests unitaires (sans base de données)

Fichier tests/ApiKernelTest.php — vérifie la logique pure du socle avec des stubs.

php tests/ApiKernelTest.php
# → Résultat : 10 réussi(s), 0 échec(s)

Testés : field(), text() (anti-XSS d'entrée), absoluteUrl(), et la sécurité de la clé (absente / mauvaise / presque correcte → 401 ; bonne → acceptée).

Tests d'intégration (contre le serveur)

Fichier tests/api_integration.ps1 — frappe l'API réelle sans créer de données (seulement des cas 401 / 422 + le ping).

powershell -File tests/api_integration.ps1
# ou : powershell -File tests/api_integration.ps1 -Base https://bokonzi.com/rpn -Key VOTRE_CLE
# → Résultat : 6 reussi(s), 0 echec(s)
